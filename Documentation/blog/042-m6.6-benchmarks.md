# M6.6: Syscall Performance Benchmarking

M6.6 expanded the benchmark suite to 28 syscalls, established a fair
Linux-under-KVM baseline, and fixed every regression except one: page
fault handling is 14% slower than Linux, traced to structural issues
in our page allocator and exception handler.

---

## Methodology

All benchmarks use KVM with `-mem-prealloc` and CPU pinning (`taskset
-c 0`) for stable measurements.  We run Linux inside the same QEMU/KVM
setup as Kevlar for a **fair comparison** — native Linux benchmarks are
misleading because they skip VMCALL/EPT overhead entirely.

Three runs per benchmark, best result taken.

## The mmap_fault investigation

`mmap_fault` benchmarks demand paging: mmap 16MB anonymous, touch all
4096 pages sequentially.  With fault-around of 16, this triggers ~256
page fault handler invocations, each allocating 17 pages (1 primary +
16 prefault).

### What we tried and why it didn't help

**Batch PTE writes** — traverse the page table once, write 16 PTEs
directly instead of 16 separate traversals.  Result: **no improvement**.
The repeated traversals hit L1 cache (same 3 intermediate entries),
so each "redundant" traverse costs ~5ns, not ~30ns.  The batch function
added its own loop overhead that canceled the savings.

**Pre-zeroed page cache** — zero pages during cache refill so
`alloc_page` returns ready-to-use pages.  Result: **broken**.
`free_pages()` pushes dirty pages back into the cache, mixing zeroed
and dirty entries.  Would need a separate clean/dirty tracking system.

**Zero hoisting** — zero all 16 prefault pages before mapping any.
Result: **worse**.  64KB of zeroing thrashes the 32KB L1 data cache,
causing the subsequent PTE writes to miss L1.  The original interleaved
zero+map keeps cache lines warm.

**Unconditional PTE writes** — skip the read-compare-branch on
intermediate page table entries.  Result: **worse**.  The unconditional
write dirties cache lines that were shared/clean, costing more than
the branch it eliminated.

**Per-CPU page cache** — eliminate spinlock contention.  Result:
**~0.6ns/page savings**.  The `lock_no_irq()` on the global cache is
just one atomic instruction (~5ns); on a single CPU it never spins.
Not the bottleneck.

### Root cause: structural overhead

The 14% gap (231ns per page) comes from three sources that require
multi-day structural changes:

**1. Bitmap allocator (~40%)** — Our allocator scans bytes in a bitmap
to find free pages (`(!byte).trailing_zeros()`).  Linux uses a buddy
allocator with O(1) free lists per order.  On cache miss, our bitmap
scan costs ~50ns vs Linux's list pop at ~5ns.  The page cache hides
this for the first 64 allocs, but refill from bitmap is the slow path.

**2. Exception handler overhead (~35%)** — Our page fault ISR saves all
16 GPRs, constructs a full `InterruptFrame`, calls through a Rust
function pointer dispatch, and checks for signal delivery on return.
Linux saves ~6 registers and jumps directly to the C handler.  This
adds ~80ns per fault (not per page — amortized over 16 fault-around
pages, it's ~5ns/page, but it's a fixed cost).

**3. Rust codegen (~25%)** — Rust generates larger function bodies than
equivalent C (enum discriminant checks, Option unwrapping, Result
propagation).  This increases instruction cache pressure in the page
fault hot path.  The effect is ~3-5ns per page from icache misses.

### What will fix it

These are tracked as M6.6 Phases 4-6:

- **Phase 4: Buddy allocator** — Replace bitmap allocator with a buddy
  allocator.  O(1) allocation, O(1) free, natural per-order free lists.
  Expected savings: ~100ns per fault (~6ns/page amortized).

- **Phase 5: Lightweight page fault entry** — Minimize register saves
  in the page fault ISR.  Save only callee-saved registers (6 instead
  of 16).  Skip signal check on fault return when no signals are pending
  (fast-path the common case).  Expected savings: ~60ns per fault.

- **Phase 6: Per-CPU page free lists** — Each CPU maintains its own
  free page list.  Allocation is a simple pointer bump with no lock.
  Refill from buddy allocator in batches of 32.  Expected savings:
  ~40ns per fault.

Combined, these should close the 231ns gap and bring mmap_fault within
10% of Linux KVM.
